## üöÄ Contributing Visual Documentation

### Creating Diagrams

**Recommended Tools:**
- **Draw.io / diagrams.net** - Free, web-based, exports SVG
- **Mermaid** - Code-based diagrams (can be in .md files too)
- **Lucidchart** - Professional diagramming
- **Excalidraw** - Hand-drawn style diagrams
- **PlantUML** - Code-based UML diagrams

### Adding a New Diagram

1. **Create the diagram** using your preferred tool
2. **Export as SVG or PNG**
3. **Name descriptively** following conventions
4. **Place in this folder** (`docs/visual/`)
5. **Reference in documentation**
   ```markdown
   ![Architecture Diagram](visual/game_architecture_diagram.svg)
   ```
6. **Update this README** with the new diagram in the index below

### Screenshot Guidelines

- Use **high resolution** (at least 1920x1080 for desktop)
- Show **clear examples** of what you're documenting
- Add **annotations** if needed to highlight features
- **Crop** to relevant areas
- **Name with date** if showing UI that might change

## üí° Tips for Visual Learners

When reading documentation:
1. **Start with diagrams** - Get the big picture first
2. **Use screenshots** - See how things actually look
3. **Follow flowcharts** - Understand processes step-by-step
4. **Refer to visual references** - Examples are clearer than text

When creating documentation:
1. **Add a diagram** whenever explaining a complex concept
2. **Include screenshots** when showing how to use something
3. **Create flowcharts** for multi-step processes
4. **Use visual hierarchies** - Organize visually with boxes, colors, arrows
